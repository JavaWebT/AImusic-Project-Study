### P5 验证码
- 作用：防止人机遍历账号密码
-  逻辑：先写一个control类，具体来说：
-  1. 调用验证码接口，生成验证码对象captcha_A（比如 带计算的图片或者是带数字的图片）
   2. 将验证码转换为文本存入变量 Code，并调用缓存将 Code 转换为一个随机生成的 CodeKey 存起来到缓存中，并为这个 CodeKey 设置了一个过期时间，防止人机爆破
   3. 将captcha 图片转换为base64码base64_captcha_A
   4. 创建一个Vo对象captcha_Vo并将（base64_captcha_A，CodeKey)保存到captcha_Vo中，返回给用户
   5. 因为 CodeKey 与 Code 是对应的，若用户传来一个数字，随后会根据用户的CodeKey来查找缓存中的Code，与其对照，实现功能

### P7登录
- Token **是一种身份验证机制，通常由服务器生成并返回给客户端，客户端在后续的请求中携带 Token，以证明自己的身份**。在 Java 中，Token 的应用场景非常广泛，例如**用户登录**、API 认证、OAuth 授权等等。
- 登录使用的是数据库中存储的MD5编码后的密码
- 登录时添加验证码校验，能够在一定程度上降低数据库的访问量
- 注册完成后，用户再登录，方便用户将账户信息记忆的更清楚

### P12.音乐播放量更新
- <img width="487" height="467" alt="image" src="https://github.com/user-attachments/assets/fc889d0e-74a7-4120-a75f-cb77b51087a2" />
- 并发性问题：更新播放量可能会出现，多人同在一个时间段点击，出现的并发问题，导致无法正确更新播放量。
- 解决方案
- 1.做法：通过直接对数据库的字段来更新，而不通过前端计算好后再更新数据库这种方式。
- 2.原因：直接对数据库字段操作，一定会按照顺序来执行，不会出现前端同一个时间段计算后，传入相同值，导致更新错误的问题

- 对于AOP的简单理解
- 1. AOP将我们要在代码中实验的业务功能与系统事务（权限管理，事务处理等）分离,提高代码的可维护性，拓展性。
  2. AOP有五种，最常使用的有 before、after 以及 around,分别是在方法调用前、后、前或后触发切片事件。
  3. AOP种的横向关注点（几乎每个业务都会触发的切片功能），连接点，切入点（多个连接点）。
